Complete Workflow

From first idea to merged code — every step, every agent, every command.

01 Project Setup one-time per project
/kit-tools:init-project
Select project type (API, Web App, Full Stack, CLI, Library, Mobile) — scaffolds the kit_tools/ directory with tailored templates.
kit_tools/ scaffold CLAUDE.md
/kit-tools:seed-project
Claude explores your codebase and fills every template with project-specific content. Runs in tiers by complexity.
generic-explorer generic-seeder ×N populated docs
02 Vision one-time or per major initiative
/kit-tools:create-vision
Guided conversation to define target users, value proposition, feature areas, constraints, and build order. Runs 3 sequential AI reviews.
vision-reviewer — completeness vision-reviewer — feasibility vision-reviewer — spec-readiness PRODUCT_VISION.md
03 Session Lifecycle every work session
/kit-tools:start-session
Orient Claude to the project state, check for orphaned notes, create scratchpad. Auto-triggered hook creates SESSION_SCRATCH.md.
SESSION_SCRATCH.md
Work happens — hooks auto-update doc timestamps, surface execution notifications, and remind to capture notes before context compaction.
/kit-tools:checkpoint mid-session
Snapshot notes mid-session without closing. Or use /kit-tools:close-session to process all notes → permanent docs and delete the scratchpad.
04 Planning per epic
/kit-tools:plan-epic
Interactive brainstorm → scope assessment → decomposition into session-fit feature specs. Everything is an epic — single-spec features get a minimal wrapper.
epic-*.md feature-*.md ×N
DECOMPOSITION EXAMPLE
epic-oauth.md ├── feature-oauth-schema.md (seq 1, no deps) ├── feature-oauth-api.md (seq 2, depends_on: schema) └── feature-oauth-ui.md (seq 3, depends_on: api)
05 Pre-Execution Validation quality gate before coding
/kit-tools:validate-epic
Four sequential agent reviews per feature spec. Each review pauses for user revision before continuing. The 4th review uses Sonnet for cross-model validation.
spec-completionist-reviewer story-quality-reviewer salty-engineer-reviewer spec-second-opinion (Sonnet)
Review 1
Are we missing any features or flows?
Review 2
Story IDs, sizing, detail, integrations
Review 3
Adversarial GAN-style engineer review
Review 4
Cross-model architecture & alternatives
↩ Revision needed? Spec is updated in-session, then reviews re-run — back to /kit-tools:plan-epic for structural changes
06 Execution per story, autonomous loop
/kit-tools:execute-epic
Select epic, choose mode, launch. The orchestrator processes each story on an isolated git branch with adaptive timeouts, pre-flight checks, structured retry context, and cross-story regression detection.
Supervised Autonomous Guarded
Per-story loop
story-implementer implements story
story-verifier checks criteria
FAIL → retry (loops back, up to max retries)
PASS → next story ↓
MONITORING
/kit-tools:execution-status tmux attach -t kit-exec-<name> touch .pause_execution
07 Post-Execution Validation auto-runs after execution
/kit-tools:validate-implementation
Four parallel review passes against the full branch diff (git diff main...HEAD). Critical findings auto-fixed, remainder logged.
code-quality-validator security-reviewer feature-compliance-reviewer feature-fixer (auto-fix) AUDIT_FINDINGS.md
08 Completion
/kit-tools:complete-implementation
Archive completed feature specs, create PR or merge to main, clean up execution artifacts.
specs/archive/ GitHub PR merge to main
↺ Next epic? Return to /kit-tools:plan-epic — Phase 4

Skills

18 skills covering the full development lifecycle — from project setup to epic completion.

/kit-tools:init-project Initialize kit_tools with project-type presets. Scaffolds the documentation structure tailored to your stack.
/kit-tools:seed-project Populate all templates from codebase exploration. Claude reads your code and fills docs with real content.
/kit-tools:seed-template Seed or re-seed a single template. Supports --force, --no-cache, --validate.
/kit-tools:validate-seeding Validate all seeded templates for unfilled placeholders and incomplete sections.
/kit-tools:create-vision Define the product vision through guided conversation with 3-pass AI review. Produces PRODUCT_VISION.md.
/kit-tools:start-session Orient Claude to the project, check for orphaned notes, create session scratchpad. Run at the top of every session.
/kit-tools:close-session Process scratchpad notes into permanent docs and delete the scratchpad. Run at the end of every session.
/kit-tools:checkpoint Mid-session checkpoint — save progress to docs without fully closing. Useful before context approaches limits.
/kit-tools:plan-epic Brainstorm and plan a new feature or epic. Creates epic-*.md wrapper plus decomposed feature-*.md specs.
/kit-tools:validate-epic Pre-execution quality gate. Four agent reviews (completeness, story quality, adversarial engineering, cross-model second opinion) per spec.
/kit-tools:execute-epic Execute an epic's feature specs in supervised, autonomous, or guarded mode. Managed by the Python orchestrator.
/kit-tools:execution-status Check progress, errors, and available actions for a running autonomous execution. Read-only — inspects state files.
/kit-tools:validate-implementation Validate the completed branch against its feature spec. Code quality, security, and compliance — 4 parallel passes.
/kit-tools:complete-implementation Mark a feature spec as completed, archive it, and handle the PR or merge. Works for mid-epic and final specs.
/kit-tools:sync-project Full sync between code and docs via drift detection. Use --quick for a lightweight audit.
/kit-tools:update-kit-tools Check if your project's hooks and templates are behind the plugin. Selectively pull in updates.
/kit-tools:optimize-tests Audit test suite health: mapping completeness, stale tests, coverage overlap, performance, and KitTools alignment. Periodic maintenance.
/kit-tools:sync-symlinks Force-refresh skill symlinks after a plugin update. Fixes autocomplete when symlinks go stale.

Automation Hooks

Eight hooks run automatically in response to Claude Code events — no commands required.

Hook Trigger What it does
create_scratchpad SessionStart Creates SESSION_SCRATCH.md if kit_tools/ exists in the project
sync_skill_symlinks SessionStart Verifies skill symlinks against installed_plugins.json — fixes stale symlinks after plugin updates
update_doc_timestamps PostToolUse (Edit/Write) Updates "Last updated" date in any kit_tools doc that Claude modifies
detect_phase_completion PostToolUse (Edit) Detects when all criteria in a feature spec are checked; suggests /kit-tools:validate-implementation
validate_seeded_template PostToolUse (Edit/Write) Checks edited templates for unfilled placeholder tokens and flags them inline
remind_scratchpad_before_compact PreCompact Reminds Claude to capture key notes before context is summarized; adds compaction marker
check_execution_notifications UserPromptSubmit Surfaces execution notifications (completions, failures, crashes, pauses) on the user's next message
remind_close_session Stop Reminds to run /kit-tools:close-session if the scratchpad has unprocessed notes

Session-Fit Guidelines

Feature specs are sized for reliable autonomous execution — one story per Claude session.

5–7 Stories per feature spec (max 7)
3–5 Criteria per story (max 6)
≤35 Total criteria per spec

Specs exceeding these limits are flagged session_ready: false and should be decomposed before execution. The validate-epic skill enforces these bounds automatically.

Philosophy

Why KitTools exists.

AI assistants are most effective when they have structured, comprehensive context about a codebase. Instead of repeatedly explaining your project in chat, maintain living documentation that gives Claude immediate understanding of architecture, patterns, and conventions.

Good documentation prevents Claude from making decisions that violate existing patterns, enables effective troubleshooting with log locations and debug procedures, and reduces errors by surfacing gotchas and known issues upfront.

The documentation also benefits human developers — but it's optimized for AI consumption.

Template vs. Instance. Plugin templates are canonical versions that update with the plugin. Your project's kit_tools/ directory is yours to customize — plugin updates never overwrite your docs. Pull in improvements selectively with /kit-tools:update-kit-tools.
